Developer Guidelines for Real-time Applications - Pilot Institute

Follow these guidelines to ensure the Unity project is optimized for real-time applications and to keep the pipeline flowing:

Documentation:

Developers will be responsible for documenting their work and will particular attention when documenting the following:

  • Layer uses (for collision, culling, and raycasting -- essentially, what should be in what layer).
  • Tag uses.
  • UI depths for layers (what should display over what).
  • Scene setup.
  • Idiom preferences.
  • Prefab structure.
  • Animation layers.
  • The organisation of your scenes, project folder, and script folder should follow a similar pattern.

Naming Conventions and coding guidelines:

Developers will aim to adhere to C# standards, and will avoid using JavaScript/UnityScript. Consequently, our coding standards will minimize the usage of hyphens (--)* and *underscores (_).

Namespaces should all be PascalCase:

For example,

Avangarde.CurrentApplication.UI.MainMenu

Classes and Methods should all be PascalCase:

For example,

public class MyClass
{
    public void SomeMethod{}
}

Fields should all be camelCase:

For example,

public static int SomeStaticValue = 10;

Properties should all be PascalCase:

For example,

private string name
public string Name
{
    get
    {
        return name;
    }
    set
    {
        name = value;
    }
};

Parameters should all be camelCase:

For example,

public void HighlightElement(bool someCondition)

Callback suffixes should be added to all delegates:

For example, when declaring a delegate type for processing a user,

public delegate void ProcessUserCallback(User u);

Declarations:

Access level modifiers should be used at all times. The compiler will infer that something is private if you don't provide an access modifier, but that doesn't imply you shouldn't use the keyword private.

For example,

private int privateVariable;

Developers should use a separate line for all variable declarations:

For example,

private int firstVariable;
private int secondVariable;

Developers should declare just one class or interface per source file. All interfaces should be prefaced with an I:

For example,

IPointerDown

Prefab use:

Developers should make full use of prefabs for scene creation and ensure that prefab connections are maintained across scenes:

Spacing in scripts:

As a rule of thumb all scripts should minimise redundant whitespace. Developers are encouraged to use Visual Studio Community's auto formatting features such as the use of the shortcut, ctrl + shift + F after selecting a block of code to tidy up tab spacing.

Insert a single blank line between methods to enhance the class's structural clarity. Similarly, include a blank line within methods to delineate distinct functionalities. For instance, if a method involves object creation and initialization, it's advisable to separate these aspects. If a method encompasses multiple logical segments, consider refactoring by creating separate methods, each responsible for a specific task.

Version Control:

Developers will use version control software, such as GIT, SourceTree, or Subversion to interact and make changes to the project repository. A code review process will be set up and adhered to before committing or merging any changes to the repository. Initial development of a feature will be done on the develop branch of the repository and merged later onto the main branch.

Testing:

Developers will be responsible for all testing and debugging of an assigned feature. For larger feature-sets a second developer should also assist with testing and debugging the functionality.

Pre-Delivery Checklist

Use this checklist to ensure your work is ready for delivery:

Your work is ready to be uploaded to the cloud, or to commit.
  • Your code is fully documented.
  • All naming conventions for variables/functions/classes are adhered to.
  • Your scene is tidy and well organised with all prefab connections maintained.
  • The feature is fully tested on device and does not interfere with any other feature's logic.
